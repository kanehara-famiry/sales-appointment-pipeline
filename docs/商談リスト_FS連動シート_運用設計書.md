# 商談リスト・FS連動シート 運用設計書（Single Source of Truth）

**バージョン**: 2.0  
**作成日**: 2026-07-11  
**更新日**: 2026-07-11  
**ステータス**: v2.0 設計書として全面リライト  
**このドキュメントの位置づけ**: 本ドキュメントは、商談リストとFS連動シートに関する**唯一の設計書**とする。「なぜこの運用が存在するのか」という思想から、「各ツールの役割」「運用ルール」「判断基準」までを整理する。  
**本ドキュメントが含まないもの**: 操作手順、Notion掲載文章、動画台本、Slack投稿文 — これらは本設計書を基準に各媒体向けに作成する。

## 1. この運用の思想・目的

### 1.1 なぜこの運用が存在するのか

フィールドセールスチームは、マネージャー（鈴木・大林）が各案件の状況把握に追われ、大きな改革を起こす余裕がない状態にあった。業務委託として招聘されたFSメンバーは「助ける側」の立場であり、自ら情報を整えることでマネージャーをクリアな状態にし、チーム全体の営業成果を高める必要がある。

しかし、当時使用していた商談リストだけでは以下の課題があった。
- シートが重く、日常的な操作に支障がある
- 入力項目を増やせない
- リアルタイムで状況共有できない
- マネージャーが案件状況を把握しにくい

本来であればCRM等で解決する内容だが、コストや導入タイミングの制約があった。そこで業務委託側より提案した改善施策としてFS連動シートが誕生した。

### 1.2 商談リストの目的

商談リストは**「業務報告・記録」のための台帳**である。

フィールドセールスの仕事は商談を「実施した」で終わりではない。会社に対して以下を報告し、記録として残すことまでが業務である。
- 何をしたのか
- 現在どういう状況なのか
- ここまでどんな対応をしたのか
- 最終的にどういう結果だったのか

すなわち、**商談リストへ記載されていない状態とは、業務報告が完了していない状態**である。

### 1.3 FS連動シートの目的

FS連動シートは**「営業成果を高めるための運用ツール」**である。

以下の目的で運用する。
- マネージャーの状況確認工数を減らす
- チーム内でフォローを強化する
- 案件の取りこぼしを防ぐ
- 営業成果を高める

現在は試験運用を経て、入力時間も正式な業務時間として認められている。

### 1.4 商談リストとFS連動シートの違い

| 観点 | 商談リスト | FS連動シート |
|------|-----------|-------------|
| 位置づけ | 会社への業務報告・記録の台帳 | 営業成果を高めるための運用ツール |
| 主な利用者 | 会社（管理職・データチーム） | FSメンバー自身・マネージャー |
| 管理対象 | 商談結果の報告・クローズ情報・最終記録 | フォロー状況・ネクストアクション・日々の活動 |
| 思想 | 報告責任を果たす場 | チームの成果を最大化する場 |

## 2. 各ツールの役割

### 2.1 商談リストが管理するもの／管理しないもの

**管理するもの**
- 商談結果の報告（企業名、最新商談日、ステータス、Memo等）
- クローズ情報（受注・失注の最終結果）
- 最終的な記録（案件の全履歴）
- 担当者（メイン担当者のみ）

**管理しないもの**
- 日々のフォロー状況
- ネクストアクション（次回コネクト予定・電話・メール等の細かいアクション）
- 案件の進捗状況の逐次更新（これらはFS連動シートで管理）

### 2.2 FS連動シートが管理するもの／管理しないもの

**管理するもの**
- フォロー状況（案件の現在の進捗状態）
- ネクストアクション（次回コネクト予定・電話・メール・見積もり等）
- 日々の活動状況（リアルタイムの更新）
- 最新商談日
- 次回コネクト予定

**管理しないもの**
- クローズ情報（商談リストで管理。クローズ後は自動的に連動シートから除外される）
- 企業名（商談リストをマスターとする）
- 担当者（商談リストをマスターとする）

### 2.3 項目ごとの正本一覧

以下の表は、各項目の情報の正本（マスター）を定義する。

| 項目 | 正本 | 転記方向 | 備考 |
|------|------|---------|------|
| 企業名 | **商談リスト** | → FS連動シート（翌日自動反映） | 商談リストへの追加が起点 |
| 担当者 | **商談リスト** | → FS連動シート | メイン担当者のみ管理 |
| 最新商談日 | **FS連動シート** | → 商談リスト（転記） | リアルタイム更新はFS連動シートで行い、転記で商談リストへ反映 |
| 次回コネクト予定 | **FS連動シート** | → 商談リスト（転記） | |
| ネクストアクション日 | **FS連動シート** | → 商談リスト（転記） | |
| フォロー状況 | **FS連動シート** | → 商談リスト（転記） | |
| 現行ステータス | **FS連動シート** | → 商談リスト（転記） | |
| 商談Memo | **FS連動シート** | → 商談リスト（転記） | |
| クローズ情報（受注/失注） | **商談リスト** | — | FS連動シート側では管理しない。クローズ後は自動除外 |
| 行ったアクション履歴 | **FS連動シート** | → 商談リスト（転記） | コピー転記ボタンの対象範囲 |

**転記される項目（FS連動シート→商談リスト）**:
行ったアクション、最新商談日、次回コネクト予定、ネクストアクション日、現行ステータス、クローズ予定 等（コピー転記ボタンの対象範囲）

**商談リストのみで管理する項目**:
企業名、担当者、クローズ情報（受注/失注）

## 3. 全体フロー（設計）

### 3.1 通常メンバーのアクションフロー

```
案件に関する行動（商談・電話・メール等）を実施
   │
   ▼
【ステップ1】FS連動シートを更新（リアルタイム入力が望ましい）
   - 最新商談日、次回コネクト予定、ネクストアクション日、現行ステータスを更新
   - 商談内容をMemoに記録
   │
   ▼
【ステップ2】1日の業務終了時までに、商談リストを更新
   - FS連動シートの内容を転記（コピー転記ボタン推奨）
   - または手動で入力
   │
   ▼
【終了条件】1日の業務終了時点で、商談リスト・FS連動シートともに最新状態であること
```

### 3.2 クローズ時のフロー

```
受注・失注が確定
   │
   ▼
【商談リスト】クロージングステータスに「受注」または「失注」を入力
   │
   ▼
【自動】FS連動シートから当該案件が自動除外される（翌々営業日程度）
   ※FS連動シート側での更新は不要
```

### 3.3 新メンバーのアクションフロー

```
【ベースキャンプ参加前（3回商談期間）】
   商談リストのみ更新（FS連動シートは利用しない）
   │
   ▼
【ベースキャンプ参加時】
   TimeRex設定・Welcome投稿などを行うタイミングでFS連動シートを付与
   │
   ▼
【以降】通常運用へ移行（通常メンバーのフローに従う）
```

## 4. 商談リスト設計

### 4.1 責任者

商談リストの更新責任者は、**案件のメイン担当者**とする。商談リストの担当者欄には、常に現在のメイン担当者を記載する。

**同席者**はサブ担当である。メイン担当が変更にならない限り、担当者は変更せず、更新責任もメイン担当者が負う。

### 4.2 更新タイミング

商談リストの更新は以下のタイミングで行うことを基本とする。

- **商談終了後**（当日の業務終了時までに反映されていればよい）
- **クローズ時**（受注・失注が確定した時点）

リアルタイム入力である必要はない。その日の業務終了時までに最新状態になっていればよい。

### 4.3 必須管理項目

- 企業名
- 担当者（メイン担当者）
- 現行ステータス
- 最新商談日
- 次回コネクト予定
- 商談Memo
- クロージングステータス（受注/失注）

### 4.4 担当者変更ルール

- メイン担当者が変更になった場合は、商談リストの担当者欄を更新する
- 同席者はサブ担当であり、メイン担当者が変わらない限り担当者欄は変更しない

### 4.5 新メンバー案件の扱い

- 3回商談終了後、担当者を新メンバーへ変更する
- フォローも新メンバーが担当する
- クローズまで新メンバーが更新を継続する

### 4.6 クローズ時の扱い

- 受注・失注確定後、クロージングステータスを入力する
- クローズ後の案件データは削除せず、履歴として保持する

## 5. FS連動シート設計

### 5.1 入力者

- **基本**: 案件を担当するFSメンバー（メイン担当者）
- **ワーカー（データチーム）**: AK列の差分セル更新を担当

### 5.2 入力タイミング

FS連動シートの更新は以下のタイミングで行うことを基本とする。

- **商談終了後**: 最新情報を反映する
- **フォロー期間中**: アクション（電話・メール受信等）が発生するたびに随時更新する

### 5.3 管理範囲（Z列〜AL列）

FS連動シートのZ列〜AL列が管理対象範囲である。この範囲で以下の情報を管理する。

- 最新商談日
- 次回コネクト予定
- ネクストアクション日
- 現行ステータス
- フォロー状況
- 行ったアクション履歴

### 5.4 リアルタイム入力の位置づけ

- **FS連動シート**: リアルタイム入力が**望ましい**（努力目標）。ただし必須ではない。
- **商談リスト**: リアルタイム入力である必要はない。
- **共通ルール**: 商談リスト・FS連動シートともに、**その日の業務終了時までに最新状態へ更新されていれば問題ない**。

### 5.5 クローズ時の扱い

クローズ時はFS連動シート側での更新は不要である。商談リストでクローズされると、自動的に連動シートから除外される。

## 6. 新メンバー運用設計

### 6.1 ベースキャンプ参加前（3回商談期間）

- FS連動シートは**利用しない**
- **商談リストのみ**更新する
- 入力練習として、新メンバー自身が商談リストへの入力を担当する

### 6.2 ベースキャンプ参加時

TimeRex設定・Welcome投稿などを行うタイミングで、FS連動シートを付与する。このタイミングをもって通常運用へ移行する。

### 6.3 通常運用移行後

- 案件担当者を新メンバーへ変更する（商談リストの担当者欄を更新）
- フォローも新メンバーが担当する
- クローズまで新メンバーが更新を継続する

## 7. 判断基準

### 7.1 差分が発生した場合

商談リストとFS連動シートの間で差分が発生した場合の判断基準は、**「項目ごとの正本一覧」（2.3節）** に従う。

- 正本がFS連動シートの項目に差分がある → FS連動シートの情報を基準として商談リストを修正する
- 正本が商談リストの項目に差分がある → 商談リストの情報を基準としてFS連動シートを修正する
- AK列（連動シート入力状況）の「差分あり」セルについては、**ワーカー（データチーム）**が更新を行う

### 7.2 担当者変更時

- メイン担当者の変更が発生した場合、商談リストの担当者欄を速やかに更新する
- 同席者はサブ担当であり、メイン担当者が変わらない限り担当者欄は変更しない
- 新メンバーへの引き継ぎ時は、3回商談終了後に担当者を変更する

### 7.3 入力遅延時

- FS連動シートへのリアルタイム入力が難しい場合でも、**その日の業務終了時までに**両シートが最新状態になっていれば問題ない
- リアルタイム入力が望ましいが、それを理由に報告や記録が遅れることのないようにする

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| **商談リスト** | 会社への業務報告・記録のための台帳。案件の最終記録とクローズ情報を保持するマスターデータ。 |
| **FS連動シート** | 営業成果を高めるための運用ツール。フォロー状況・ネクストアクション・日々の活動を管理する。別称「連動シート」。 |
| **転記** | FS連動シートの内容を商談リストに反映する作業。コピー転記ボタンでのワンクリック転記が推奨される。 |
| **コネクト** | 以下の3つのいずれかを満たすこと：①担当者と電話ができた ②担当者からメールを受信した ③2回目以降の商談。※1回目の商談は対象外。こちらから送信したメール（送信）は対象外。 |
| **最新商談日（最新コネクト日）** | その案件について最後にコネクト（上記定義を満たすアクション）を行った日付。 |
| **次回コネクト予定** | 次に顧客とコネクトする予定の日付。ネクストアクション日とは別物。 |
| **ネクストアクション日** | 次に取るべきアクション全体の期限日。電話・打ち合わせ・見積もり・新規カテゴリ登録など、すべてのアクションを含む。次回コネクト予定と一致する場合もあれば一致しない場合もある。 |
| **現行ステータス** | 案件の進捗度合いを示す％表記（例：10%, 20%, 40%, 50%, 80%等）。全件が入力対象。 |
| **クロージングステータス** | 案件の最終結果。「受注」または「失注」。商談リストで管理し、FS連動シートには反映されない。 |
| **AK列** | FS連動シート入力状況シート内の、差分の有無を示す列。ワーカー（データチーム）が更新を行う。 |
| **メイン担当者** | 案件の更新責任を負うFSメンバー。商談リストの担当者欄に記載される。 |
| **同席者（サブ担当）** | 商談に同席したFSメンバー。メイン担当者が変わらない限り、担当者欄は変更せず、更新責任もメイン担当者が負う。 |
| **ベースキャンプ** | 新メンバーの研修・育成プログラム。参加前にFS連動シートは付与されない。 |

## 9. 例外・未確定事項

以下は、現時点で判断できない内容である。鈴木・大林への確認が必要。

### 9.1 転記の頻度・タイミング

- ワーカー（データチーム）が転記作業を実施する頻度（毎日？週次？）
- 転記の締め切りタイミング
- 転記実施者が寺西以外にいるか

### 9.2 現行ステータス（％）の具体的定義

- 10%, 20%, 40%, 50%, 80%等がそれぞれ何を意味するのか
- ステータス遷移のルール（何をトリガーに上げるのか）
- 受注時は「100%＋クロージングステータス」なのか

### 9.3 引き継ぎの具体的プロセス

- 旧担当者から新担当者（新メンバー）への引き継ぎ手順
- 引き継ぎの確認方法（誰が承認するのか）
- 引き継ぎ漏れが発生した場合の対応

### 9.4 商談リストと連動シートのURL

- 実際のスプレッドシートのURL（アクセス権限含む）

### 9.5 「100%入力」との関係性

- 本運用と「100%記入」の運用はどのように接続するのか
- 100%記入はスコープ外だが、連動シートの全件入力との境界線を明確にする必要がある

## 補足：本設計書の使い方

1. **設計書としての位置づけ**: 本ドキュメントはルール集ではない。「なぜこの運用が存在するのか」という思想から、「各ツールの役割」「運用ルール」「判断基準」までを定義した設計書である。
2. **媒体作成時の基準**: Notionマニュアル、動画、Slack投稿を作成・改訂する際は、必ず本設計書を基準とすること。操作手順・掲載文章・台本等は本設計書から各媒体向けに作成する。
3. **更新ルール**: 本設計書に変更が生じた場合は、版管理を行い変更履歴を残すこと。
4. **確認事項の解決**: #9 の未確定事項が解決したら、本設計書をアップデートしバージョンを上げること。
