## 1. この運用の目的

商談リストとFS連動シートを運用する目的は、以下のとおりである。

1. **マネージャーのモグラたたき状態からの脱却**  
   これまでマネージャー（鈴木・大林）は、各案件の状況把握に追われ、大きな改革を起こす余裕がなかった。FSメンバー（業務委託）は「助ける側」として、自ら情報を整えることでマネージャーをクリアな状態にする。

2. **案件情報の可視化と一元管理**  
   FSメンバーおよび管理職が、各案件の進行状況・ステータス・次回アクションをいつでも確認できる状態にする。

3. **データの正確性と連動の維持**  
   商談リストとFS連動シートの間でデータを転記・同期することで、二重管理による情報の乖離を防止する。

4. **営業プロセスの標準化**  
   商談終了後の入力・更新フローを標準化し、メンバーによらず一貫した運用を実現する。

5. **マネジメントの判断材料**  
   受注確度・案件分布・メンバー稼働などのデータを蓄積し、経営判断・リソース配分の基礎とする。

## 2. 全体フロー

商談終了後から受注・失注までの、時系列の処理フローを示す。

### 2.1 標準フロー

```
【毎朝】ハットリさんから連絡
   「今日コネクトすべき案件」「今日のネクストアクション」が通知される
   │
   ▼
【日中】FS連動シートをリアルタイム更新（推奨）
   アクション発生ごとに、FS連動シート（Z〜AL列）へ入力
   - 電話をかける → 電話中または直後に更新
   - メール受信 → 受信時に更新
   - 商談実施 → 終了後に更新
   │
   ▼
【日次 or 随時】商談リストへの転記
   方法A（推奨）: コピー転記ボタンでワンクリック転記
   方法B: 手動で商談リストを開いて入力
   ※商談リストは1日1回の入力でもOK（重い・リスク有のため）
   │
   ▼
【ステータス更新（随時）】
   案件の進展（追加商談・受注・失注）に応じて、都度両シートを更新
   │
   ▼
【受注 or 失注】
   - クロージングステータスに「受注」または「失注」を入力
   - 入力後、翌々営業日程度でFS連動シートから自動消去（データは保存）
```

### 2.2 各担当者の責務

| 担当 | やること | タイミング |
|------|----------|------------|
| ハットリさん | 朝一の連絡（今日のコネクト/ネクストアクション通知） | 毎朝 |
| FSメンバー（同席者） | FS連動シート（Z〜AL列）への入力・更新 | アクション発生ごと（推奨） |
| FSメンバー（同席者） | 商談リストへの転記（コピー転記ボタン or 手動） | 1日1回以上 |
| データチーム（寺西） | FS連動シート↔商談リストの転記・差分確認 | 定期的（頻度は運用による） |
| 管理職（鈴木・大林） | 運用状況のモニタリング・是正指示 | 適宜 |

## 3. 商談リスト仕様

### 3.1 役割

商談リストは、**全案件の一次情報を保持するマスターデータ**である。  
ただし、日常の参照・更新はFS連動シートを主とし、商談リストは転記と最終的なデータ保存の基盤として機能する。

### 3.2 入力する人

- **基本**: 商談に同席したFSメンバー（1人～複数）
- **新メンバー**: 3商談目以降は新メンバー自身が入力（練習兼ねる）
- **例外**: 同席メンバーから特別な指示があった場合はその指示に従う
- **データチーム（寺西）**: 転記作業時に一部修正を行う場合あり

### 3.3 入力タイミング

- **基本**: 1日1回の記入でOK（動画でも「一日の終わりに一回記入」で良いと明言）
- **リアルタイム入力もOK**: リアルタイムで入力してもよい（やる人を否定しない）
- **更新**: 案件の進展（追加商談・受注・失注）が発生した時点で随時更新する

### 3.4 必須事項

以下の項目は、商談リストへの必須入力項目とする。

- 企業名 / 案件名
- ステータス（現行ステータス。％表記）
- 最新商談日（コネクトが発生した最新日付）
- 次回コネクト予定（またはネクストアクション日）
- 担当者
- 商談Memo（内容の簡易記録）
- クロージングステータス（受注・失注確定時）

### 3.5 更新ルール

- 新しい商談が発生するたびに、当該案件の行を**上書き更新**する（複数行追加ではない）
- ステータスは商談の進展に応じて都度更新する
- 過去の商談内容はMemo欄に残す（削除しない）

### 3.6 担当者変更ルール

- 担当者名の変更は**可能**
- 特に新メンバーが3商談を経て案件を引き継ぐ場合は、商談リストの担当者名を新メンバーに書き換える
- 変更の際は旧担当者から新担当者への**引き継ぎ**を実施すること（具体的手順は要確認）

### 3.7 新メンバー案件の扱い

- 新メンバーは**3商談目から**商談リストへの入力を開始する
- 同席メンバーからの特別指示がない限り、新メンバーが**自身で入力を担当**する
- 案件のフォローも**新メンバー自身**が行う
- 商談リストの担当者名は**新メンバーへ変更してよい**

### 3.8 受注・失注時の扱い

- **受注・失注確定後**: 商談リストのクロージングステータスに「受注」または「失注」を入力する
- クロージングステータスが入力されると、**翌々営業日程度**でFS連動シートから自動消去される（データは裏で保持）
- 受注・失注後も案件データは削除せず、履歴として残す

## 4. FS連動シート仕様

### 4.1 役割

FS連動シートは、**日常の運用でFSメンバーが主に使用するインターフェース**である。  
理想的な状態では「FS連動シートだけ開けば良い」状態を目指し、商談リストは転記時のみ開く。

### 4.2 入力する人

- **FSメンバー（同席者）**: アクション発生ごとに入力する（第一入力者）
- **データチーム（寺西）**: 転記作業時に参照・一部修正
- **管理職**: チーム全体の進捗確認用

### 4.3 入力タイミング

- **基本**: アクション（コネクト・メール受信・ネクストアクション）発生後、**リアルタイムでの入力が強く推奨される**
- リアルタイム入力が難しい場合は、まとめて入力してもよいが、転記タイミングとの兼ね合いに注意する

### 4.4 必須入力範囲

**Z列〜AL列の全て**が入力対象である。

具体的なカラム（動画から推定）:
- Z列〜AC列: 基本情報・ステータス系（未入力だと赤くなる）
- 行ったアクション 〜 クローズ予定: コピー転記対象範囲
- 最新商談日（コネクト日）
- 次回コネクト予定
- ネクストアクション日
- 現行ステータス
- クロージングステータス

※以前は40%以上の案件のみが入力対象だったが、**現在は全件（10%・20%含むすべて）が入力対象**である。

### 4.5 更新ルール

- 案件の進展に応じて、都度FS連動シートを更新する
- **商談リスト側が最新なら、連動シートもそれに合わせて更新する**
- 連動シートと商談リストの間でコピー転記ボタンによる同期が可能

### 4.6 商談リストとの関係

- 商談リストが**一次情報の保存先（マスター）**
- FS連動シートは**日常の操作インターフェース**
- 情報の流れ: **FSメンバーのアクション → FS連動シート（リアルタイム入力） → コピー転記 → 商談リスト**
- 商談リストに企業が追加されると、**翌日（午前2時ごろ）**にFS連動シートに自動追加される
- FS連動シートの並び順: **現行ステータスが高い順 → 最新商談日が新しい順**

### 4.7 差分が発生した場合の考え方

- 商談リストとFS連動シートで差分が発生した場合の**基準 = 商談リストを正とする**
- AK列（連動シート入力状況）の「差分あり」は転記時に確認する指標
- 連動シート側で未入力の項目は赤くなる（ただし赤くてもNGとは限らない。案件の性質による）
- 商談リスト側の内容と一致していない場合も赤くなる
- **寺西からの指摘**: 商談リスト側で受注になっているのに連動シート側のステータスが未更新の場合 → **商談リストを正とし**、連動シート側を修正する
- 転記時に連動シートの古い情報で商談リストを上書きしないよう注意する

### 4.8 リアルタイム入力の位置づけ

- **FS連動シート**: 【推奨】リアルタイム入力が強く推奨される。実際に制約率の高いメンバーはリアルタイム入力の習慣がある
- **商談リスト**: 【OK】1日1回の記入で問題ない（リアルタイムでもよい）
- リアルタイム入力により、マネージャーやリーダーの連動感が向上し、チーム全体の状況把握が円滑になる

## 5. 商談リストとFS連動シートの関係

### 5.1 なぜ両方必要なのか

- **商談リスト**: 全案件の一次情報を保持するマスターデータ。ただし重く、アクセス集中でダウンするリスクがあり、誰かが列を消す等の大きな影響が組織全体に及ぶ可能性がある
- **FS連動シート**: FSメンバーが日常的に使う軽量インターフェース。商談リストのリスクを軽減しつつ、リアルタイム入力を促進する
- 両者が存在することで、**FSメンバーの入力負荷軽減**と**データの完全性維持**を両立している

### 5.2 何が管理対象なのか

- **管理対象**: FSメンバーが担当するすべての商談案件
- **対象外（今回のスコープ外）**: 100%入力に関する運用改善・設計

### 5.3 重複している情報

両シートで重複して管理される情報：
- 企業名
- 現行ステータス（％表記）
- 最新商談日（最新コネクト日）
- 次回コネクト予定
- ネクストアクション日
- 担当者
- 商談Memo
- クロージングステータス（受注/失注）

### 5.4 どちらを基準にするのか

**基準（マスター）は商談リストとする。**

- 商談リストが一次情報の保存先
- FS連動シートは日常の操作画面
- 両者に差分が生じた場合は、**商談リストの情報を優先**して連動シートを修正する
- **補足**: 日常の参照・更新は**FS連動シートを主とすること**が推奨される（商談リストは重く、リスクもあるため）。FS連動シートだけ開けば良い状態が理想

## 6. 新メンバー運用

### 6.1 いつから商談リストを扱うか

- **3商談目から**商談リストへの入力を開始する
- これは新メンバー教育の一環として、実際の案件を使った練習を兼ねる

### 6.2 いつからFS連動シートを扱うか

- **現時点では未確定**
- Slack上の議論では「3商談の時点では連動シートはないはず」という金原の発言あり
- ベースキャンプニュービーの段階で連動シートを持つことの是非について検討中
- **→ 要確認事項**

### 6.3 案件担当者

- 商談に**同席したFSメンバー**が一次的な案件担当者となる
- 新メンバーが3商談を経た案件は、**新メンバー自身**が案件担当者となる
- 同席メンバーからの特別指示がある場合はそれに従う

### 6.4 フォロー担当者

- **基本**: 案件担当者 = フォロー担当者
- 新メンバーが担当する案件のフォローも、**新メンバー自身**が行う
- （大林確認済み:「同席メンバーからの特別指示がなければ新メンバー側が入力を想定」）

### 6.5 引き継ぎタイミング

- 新メンバーへの案件引き継ぎは、**3商談実施後**を基本とする
- 引き継ぎの具体的な手順（申し送り事項・実施方法）は**要確認**
- 引き継ぎ時に商談リストの担当者名を新メンバーに変更する

## 7. 必須ルールと推奨運用

### 【必須】

1. **FS連動シート（Z列〜AL列）の全件入力**
   - 10%、20%など、すべてのステータスの案件について入力する（40%以上のみの対象は過去のルール）
2. **商談リストへの転記を1日1回以上行う**
   - コピー転記ボタンを使用し、FS連動シートの内容を商談リストに反映する
3. **ステータスの更新**は案件の進展（追加商談・受注・失注）が発生するたびに行う
4. **商談リストを一次情報（マスター）** として扱う
5. 商談リストと連動シートに差分が生じた場合は、**商談リストの情報を基準**に連動シートを修正する
6. 新メンバーは**3商談目から商談リストへの入力を開始**する
7. 新メンバーが案件を担当する場合、**商談リストの担当者名を新メンバーに変更する**
8. **受注・失注確定時**はクロージングステータスに「受注」または「失注」を入力する

### 【推奨】

1. **FS連動シートへのリアルタイム入力**
   - アクション（電話・メール受信・商談）が発生したら、その場でFS連動シートを更新する
   - 電話をかけるときはFS連動シートを開きっぱなしにし、通話中または通話直後に更新する
2. **お気に入り登録**
   - FS連動シートの自分のビューをお気に入りに登録し、ワンタッチで開けるようにする
   - 商談リストはお気に入りから消し、FS連動シートを開く動線を一本化する
3. **商談リストは極力開かない**
   - FS連動シートだけで完結させる習慣をつける
   - 商談リストは重く、組織全体に影響を与えるリスクがあるため
4. **寺西からの指摘「AK列の差分あり確認」に日頃から目を配る**
5. **差分を発見した場合は放置せず修正する**
6. **コピー転記ボタンの活用**
   - 「行ったアクション〜クローズ予定」をコピーし、転記ボタンで商談リストに貼り付ける
   - 転記後は白くなっていることを確認する
7. **問い合わせ先はハットリさん**
   - 運用に関する質問は各チームのハットリさんへ

## 8. 用語定義

本仕様書で使用する用語を以下に定義する。

| 用語 | 定義 | 備考 |
|------|------|------|
| **商談リスト** | FSメンバーが案件の一次情報を記録・管理するスプレッドシート。全案件の最新ステータスを保持するマスターデータ。 | Google Sheets上に存在。重く、アクセス集中で落ちるリスクあり |
| **FS連動シート** | 商談リストとは別に存在する日常操作用シート。Z〜AL列が入力対象。FSメンバーが主に使用するインターフェース。 | 別称「連動シート」「FSレンドシート」 |
| **転記** | FS連動シートの内容を商談リストに反映する作業。コピー転記ボタンでのワンクリック転記が推奨される。 | 寺西（Data）も実施 |
| **コネクト** | 以下の3つのいずれかを満たすこと：①担当者と電話ができた ②担当者からメールを受信した ③2回目以降の商談 ※1回目の商談は対象外。こちらからメールを送信した（送信）は受信ではないので対象外。 | 動画で明確に定義 |
| **最新商談日（最新コネクト日）** | その案件について最後にコネクト（上記定義を満たすアクション）を行った日付。1回だけの商談ならその日、2回目以降なら最新の日付。 |  |
| **次回コネクト予定** | 次に顧客とコネクトする予定の日付。ネクストアクション日とは別物。コネクトする予定がある場合のみ記入。 |  |
| **ネクストアクション日** | 次に取るべきアクション全体の期限日。電話・打ち合わせ・見積もり送付・新規カテゴリ登録など、すべてのアクションを含む。次回コネクト予定と一致する場合もあれば一致しない場合もある。 | 次回コネクト予定とは明確に別物 |
| **現行ステータス** | 案件の進捗度合いを示す％表記（例：10%, 20%, 40%, 50%, 80%など）。全件入力対象。並び順は「現行ステータスが高い順 → 最新商談日が新しい順」。 | ％ごとの定義は運用基準による |
| **クロージングステータス** | 案件の最終結果。「受注」または「失注」を入力。入力後、翌々営業日程度でFS連動シートから自動消去される（データは保持）。 |  |
| **AK列** | FS連動シート入力状況シート内の、差分の有無を示す列。「差分あり」は商談リストと連動シートで情報が一致していないことを示す。 | 寺西が転記時に確認 |
| **コピー転記ボタン** | FS連動シート上のボタン。「行ったアクション〜クローズ予定」をコピーし、商談リストの該当セルに自動で飛んで貼り付ける機能。 | 転記作業の効率化ツール |
| **ハットリさん** | 各チームに存在するFS連動シートの問い合わせ窓口（担当者名）。朝一のコネクト・ネクストアクション連絡も行う。運用に関する質問はすべてハットリさんへ。 |  |
| **ベースキャンプ** | 新メンバーの研修・育成プログラムの名称。 |  |
| **BCニュービー** | ベースキャンプ内の初期段階のメンバー区分。この段階では連動シートは扱わない可能性が高い。**→ 要確認** |  |

## 9. 例外・未確定事項（要確認事項）

### 9.1 新メンバーのFS連動シート開始タイミング

- 新メンバーがいつからFS連動シートを扱い始めるのか
- BCニュービーの段階では連動シートは扱わないという認識で良いか
- それとも新メンバーも3商談目から同時に連動シートを扱うのか

- 引き継ぎ漏れが発生した場合の対応

### 9.3 転記の頻度・タイミング

## 補足：本仕様書の使い方

1. **仕様書としての位置づけ**: 本ドキュメントはマニュアルではない。あくまで「どう運用すべきか」を定義した仕様書である。
2. **マニュアル作成時の基準**: Notion、動画、Slack投稿を作成・改訂する際は、必ず本仕様書を基準とすること。
3. **更新ルール**: 本仕様書に変更が生じた場合は、版管理を行い変更履歴を残すこと。
4. **確認事項の解決**: #9 の要確認事項が解決したら、本仕様書をアップデートしバージョンを上げること。
